LangGraph has two streaming features for different levels of control:

- **Projection Streaming** is the recommended model for most application code, especially frontend application streaming. It exposes typed projections such as `run.messages`, `run.values`, `run.subgraphs`, `run.output`, and `run.extensions`.
- **Mode Streaming** is a low-level streaming primitive based on Pregel stream modes such as `updates`, `values`, `messages`, `custom`, `checkpoints`, `tasks`, or `debug`. Use it when you need access to LangGraph internals or advanced stream mode output.

Use Projection Streaming when you want ergonomic application-facing streams. Use Mode Streaming when you need lower-level access to the Pregel execution stream.

## How the pieces fit together

The streaming stack has five layers:

1. The LangGraph Pregel engine emits data based on the configured stream mode.
2. The protocol defines event envelopes, channels, namespaces, ordering, replay, and transport behavior.
3. LangGraph produces protocol events from graph execution.
4. Stream transformers build typed projections over those events.
5. SDK and frontend clients consume the same events over remote transports.

In-process streaming and remote streaming use the same event model. The difference is how the events are delivered:

:::python
- In process, Projection Streaming uses `graph.stream_v2()` to return a run object backed by Python iterables and projection properties.
- Remotely, the Python SDK currently streams deployed runs through `client.runs.stream(...)`.
:::

:::js
- In process, Projection Streaming uses `graph.stream_v2()` to return a run object backed by async iterables.
- Remotely, `client.threads.stream(...)` returns a thread stream backed by SSE, WebSocket, or a custom transport.
:::
