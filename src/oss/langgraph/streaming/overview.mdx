LangGraph has two streaming features for different levels of control:

- **Event Streaming** is the recommended model for most application code, especially frontend application streaming. It exposes typed projections such as `run.messages`, `run.values`, `run.subgraphs`, `run.output`, and `run.extensions`.
- **Pregel Streaming** is a low-level streaming primitive based on Pregel stream modes such as `updates`, `values`, `messages`, `custom`, `checkpoints`, `tasks`, or `debug`. Use it when you need access to LangGraph internals or advanced stream mode output.

Use Event Streaming when you want ergonomic application-facing streams. Use Pregel Streaming when you need lower-level access to the Pregel execution stream.

## Choose a streaming API

| Need | Read |
| ---- | ---- |

| Add app-specific projections | [Custom transformers](/oss/langgraph/streaming/custom-transformers) |
| Create local or remote custom stream projections | [StreamChannel](/oss/langgraph/streaming/streamchannel) |

## How the pieces fit together

The streaming stack has two main layers:

1. **Pregel Streaming** emits raw graph execution events from the Pregel engine.
2. **Event Streaming** normalizes those events, runs them through stream transformers, and exposes typed projections.

The event router is the bridge between the two layers. It receives normalized Pregel events and passes each event through the registered stream transformers. Built-in transformers create standard projections such as `run.messages`, `run.values`, `run.subgraphs`, and `run.output`. Custom transformers can add application-specific projections under `run.extensions`.

You can stream at either layer:

- Use [Event Streaming](/oss/langgraph/streaming/event-streaming) when you want projected events and typed stream accessors for application code.
- Use [Pregel Streaming](/oss/langgraph/streaming) when you want raw Pregel stream modes and are comfortable handling graph-runtime details directly.

In-process streaming and remote streaming use the same event model. The difference is how the events are delivered:

:::python
- In process, Event Streaming uses `graph.stream_events(..., version="v3")` to return a run object backed by Python iterables and projection properties.
- Remotely, the Python SDK currently streams deployed runs through `client.runs.stream(...)`.
:::

:::js
- In process, Event Streaming uses `graph.streamEvents(..., { version: "v3" })` to return a run object backed by async iterables.
- Remotely, `client.threads.stream(...)` returns a thread stream backed by SSE, WebSocket, or a custom transport.
:::
