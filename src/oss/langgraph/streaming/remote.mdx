Use the LangGraph SDK to stream graphs and agents that run behind a LangGraph API server.

:::python
The Python SDK currently streams deployed runs with `client.runs.stream(...)`.
:::

:::js
The recommended remote streaming entry point is `client.threads.stream(...)`.
:::

:::python
```py
from langgraph_sdk import get_client

client = get_client(url="http://localhost:2024")
thread = await client.threads.create()

async for chunk in client.runs.stream(
    thread["thread_id"],
    "simple-tool-graph",
    input={
        "messages": [{"role": "user", "content": "What is 42 * 17?"}],
    },
    stream_mode="messages-tuple",
):
    if chunk.event == "messages":
        message_chunk, metadata = chunk.data
        print(message_chunk.content, end="", flush=True)
```
:::

:::js
```ts
import { Client } from "@langchain/langgraph-sdk";

const client = new Client({ apiUrl: "http://localhost:2024" });

const thread = client.threads.stream({
  assistantId: "simple-tool-graph",
});

await thread.run.input({
  input: {
    messages: [{ role: "user", content: "What is 42 * 17?" }],
  },
});

for await (const message of thread.messages) {
  for await (const token of message.text) {
    process.stdout.write(token);
  }
}

console.log(await thread.output);
await thread.close();
```
:::

## Thread streams

:::python
Python remote streaming currently uses run stream chunks. Use `client.runs.stream(...)` with `stream_mode` values such as `updates`, `values`, `messages-tuple`, and `custom`.
:::

:::js
`client.threads.stream(...)` returns a `ThreadStream`. A thread stream is a remote handle that mirrors the in-process Event Streaming model:

| Getter | Channel | Use |
| ------ | ------- | --- |
| `thread.messages` | `messages` | Stream message text, reasoning, usage, and blocks. |
| `thread.values` | `values` | Stream state snapshots and await the final state. |
| `thread.output` | `values` | Await final output. |
| `thread.toolCalls` | `tools` | Observe tool call lifecycle and results. |
| `thread.subgraphs` | `lifecycle` | Observe nested graph execution. |
| `thread.subagents` | `lifecycle` and `tools` | Observe Deep Agents subagent work. |
| `thread.extensions.<name>` | `custom:<name>` | Consume custom transformer projections. |

The SDK opens subscriptions lazily. If a client only reads `thread.output`, it does not subscribe to token-level messages.
:::

## Submit work

:::python
Submit work when calling `client.runs.stream(...)`:

```py
async for chunk in client.runs.stream(
    thread["thread_id"],
    "my-agent",
    input={"messages": [{"role": "user", "content": "hello"}]},
    metadata={"source": "docs"},
    stream_mode="updates",
):
    print(chunk.data)
```
:::

:::js
Use `thread.run.input(...)` to start a run or resume work on the bound thread:

```ts
await thread.run.input({
  input: { messages: [{ role: "user", content: "hello" }] },
  metadata: { source: "docs" },
});
```
:::

For human-in-the-loop flows, use `thread.input.respond(...)` after an interrupt. See the frontend and SDK interrupt guides for UI patterns.

## Subscribe to raw protocol events

Most applications should use Event Streaming through typed projections. Use raw subscriptions for debugging, advanced routing, or custom rendering:

:::python
```py
async for chunk in client.runs.stream(
    thread["thread_id"],
    "my-agent",
    input={"messages": [{"role": "user", "content": "hello"}]},
    stream_mode=["updates", "messages-tuple", "custom"],
):
    print(chunk.event, chunk.data)
```
:::

:::js
```ts
const events = await thread.subscribe({
  channels: ["messages", "tools", "values", "lifecycle"],
  namespaces: [["researcher", "tools"]],
  depth: 0,
});

for await (const event of events) {
  console.log(event.method, event.params.namespace, event.params.data);
}
```
:::

Use `namespaces` and `depth` to subscribe to specific graph, subgraph, and task events. See [Namespaces](/oss/langgraph/streaming/namespaces) for matching rules and recipes.

## Reconnect and replay

Remote streams carry monotonic sequence numbers. The SDK tracks ordering, deduplicates replayed events, and replays buffered server events when a subscription reconnects.

The important application-level rule is to keep a thread ID when you need to reattach:

:::python
```py
async for chunk in client.runs.stream(
    "thread-id",
    "simple-tool-graph",
    input={"messages": [{"role": "user", "content": "continue"}]},
    stream_mode="updates",
):
    print(chunk.data)
```
:::

:::js
```ts
const thread = client.threads.stream("thread-id", {
  assistantId: "simple-tool-graph",
});
```
:::

## Run streaming APIs

:::python
`client.runs.stream(...)` is the current Python SDK streaming API.
:::

:::js
`client.runs.stream(...)`, `client.runs.joinStream(...)`, and `client.threads.joinStream(...)` continue to work for existing code. Use `client.threads.stream(...)` when you want a remote API that maps directly to the Event Streaming protocol and typed projections.
:::
